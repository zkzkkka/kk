# 毕设项目运行步骤

## 1. 项目简介

本项目是一个基于强化学习的 SDN 路由实验系统，运行依赖以下组件协同工作：

- `Ryu`：作为 SDN 控制器，负责交换机连接、链路发现、状态采集和流表下发。
- `Mininet`：用于构建实验网络拓扑。
- `PyTorch`：用于强化学习智能体的决策与在线训练。
- `Matplotlib`：用于生成实验结果图表。

项目根目录如下：

```text
/home/ubuntu/毕设
├── topology/            # Mininet 拓扑
├── control_plane/       # Ryu 控制器与监控模块
├── agent/               # 强化学习智能体
├── visualization/       # 绘图脚本
└── .venv/               # Python 虚拟环境
```

## 2. 运行环境

建议在 Ubuntu 或其他 Linux 环境中运行。本项目不适合直接在普通 Windows 环境下执行。

运行前请确保以下组件可用：

- Python 3
- Mininet
- Open vSwitch
- 项目虚拟环境中的 Python 依赖

如果需要检查关键命令是否存在，可以执行：

```bash
command -v mn
command -v ovs-vsctl
command -v /home/ubuntu/毕设/.venv/bin/ryu-manager
```

## 3. 启动前准备

每次运行前，建议先清理 Mininet 旧环境，避免残留拓扑影响实验：

```bash
sudo mn -c
```

进入项目目录：

```bash
cd /home/ubuntu/毕设
```

激活虚拟环境：

```bash
source .venv/bin/activate
```

## 4. 系统运行顺序

本系统必须按以下顺序启动：

1. 启动 `Ryu` 控制器。
2. 启动 `Mininet` 拓扑。
3. 在 `Mininet CLI` 中发起测试流量。
4. 观察控制器日志和实验结果。

## 5. 具体运行步骤

### 5.1 启动控制器

打开第一个终端，执行：

```bash
cd /home/ubuntu/毕设
source .venv/bin/activate
TOPOLOGY_SCENARIO=multiflow ryu-manager control_plane/ryu_main.py control_plane/network_monitor.py
```

说明：

- `ryu_main.py` 是控制器主程序。
- `network_monitor.py` 负责定期采集端口统计信息。
- `ryu_main.py` 内部已经启用了链路发现，因此这里不需要额外再传 `--observe-links`。
- 控制器启动后会以前台方式持续运行，等待交换机连接；终端不返回到命令提示符属于正常现象，不是“卡死”。
- 当第二个终端里的 `Mininet` 拓扑启动并连上控制器后，控制器日志才会继续出现交换机和链路相关输出。
- 推荐场景为 `multiflow`，会加载 4 主机多流竞争实验配置。
- 如果希望把实验难度再提高一档、让 RL 与 static 的差异更明显，可使用 `TOPOLOGY_SCENARIO=complexflow`。
- 默认运行模式为强化学习路由，控制器会在 `/home/ubuntu/毕设/metrics/` 下自动记录真实实验日志。
- RL 模式现在会默认使用 `/home/ubuntu/毕设/checkpoints/<scenario>_rl_agent.pt` 持久化模型、优化器、epsilon 和经验回放池，便于跨多轮实验持续训练。
- 通过 `RL_AGENT_MODE=train|eval` 可以在“持续训练”和“纯评估”之间切换。

三种场景的控制器启动命令如下。

`legacy`：

```bash
cd /home/ubuntu/毕设
source .venv/bin/activate
TOPOLOGY_SCENARIO=legacy ryu-manager control_plane/ryu_main.py control_plane/network_monitor.py
```

`multiflow`：

```bash

source .venv/bin/activate
TOPOLOGY_SCENARIO=multiflow ryu-manager control_plane/ryu_main.py control_plane/network_monitor.py
```

`complexflow`：

```bash
cd /home/ubuntu/毕设
source .venv/bin/activate
TOPOLOGY_SCENARIO=complexflow ryu-manager control_plane/ryu_main.py control_plane/network_monitor.py
```

如果需要跑一组“静态链路权重”基线实验，可使用对应场景的静态模式命令。

`legacy`：

```bash
cd /home/ubuntu/毕设
source .venv/bin/activate
```

`multiflow`：

```bash

source .venv/bin/activate
TOPOLOGY_SCENARIO=multiflow RL_ROUTING_MODE=static ryu-manager control_plane/ryu_main.py control_plane/network_monitor.py
```


### 5.2 启动 Mininet 拓扑

打开第二个终端，执行：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario multiflow
```

说明：

- 推荐场景会创建 6 台交换机和 4 台主机：`h1@S1`、`h2@S6`、`h3@S2`、`h4@S5`。
- `complexflow` 场景会在此基础上扩展为 6 台主机，并增加 `s1-s4`、`s2-s5`、`s3-s6` 三条额外核心链路。
- 拓扑中的远程控制器地址固定为 `127.0.0.1:6653`。
- 脚本会先等待交换机连上控制器、等待链路发现稳定，再自动做一轮“主机发现与流表安装预热”。
- 启动完成后会进入 `Mininet CLI`。
- 三种场景的拓扑启动命令如下。

`legacy`：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario legacy
```

`multiflow`：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario multiflow
```

`complexflow`：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario complexflow
```

### 5.3 在 Mininet 中测试网络

在 `Mininet CLI` 中先进行基础连通性测试：

```bash
pingall
```

说明：

- 由于拓扑脚本已经在进入 CLI 前自动做过预热，正常情况下第一次 `pingall` 就应该能够直接连通。
- 如果启动时看到 `Unable to contact the remote controller at 127.0.0.1:6653`，说明控制器监听端口还没完全起来；这种情况下先退出当前 Mininet，再等控制器启动完成后重新执行 `sudo python3 topology/mininet_topo.py`。

随后发起带宽测试流量：

```bash
h2 iperf -s &
h1 iperf -c 10.0.0.2 -t 20
```

说明：

- 上面是两条命令，不是一条命令。
- 先执行 `h2 iperf -s &`，在 `h2` 上启动 `iperf` 服务端并放到后台。
- 再执行 `h1 iperf -c 10.0.0.2 -t 20`，在 `h1` 上发起 20 秒测试流量。

如果希望反复测试，也可以多次执行：

```bash
h1 ping 10.0.0.2
h1 iperf -c 10.0.0.2 -t 10
```

### 5.4 动态链路实验

如果需要验证“链路故障/恢复后控制器是否能自动改路”，推荐优先使用新的 `multiflow` 多流竞争场景。

最省事的一键方式是：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario multiflow --auto-dynamic-demo --exit-after-demo
```

该命令会自动启动 4 条业务流，并在业务运行过程中对 `s4-s6` 执行一次 `down/up`。

三种场景的一键自动动态实验命令如下。

`legacy`：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario legacy --auto-dynamic-demo --exit-after-demo
```

`multiflow`：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario multiflow --auto-dynamic-demo --exit-after-demo
```

`complexflow`：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py --scenario complexflow --auto-dynamic-demo --exit-after-demo
```

说明：

- `legacy` 适合保留原始 2 主机闭环演示。
- `multiflow` 适合作为默认 RL 训练与对比实验场景。
- `complexflow` 适合放大学习信号、让 RL 与 static 的差异更容易显现。

如果你希望在 `Mininet CLI` 中手动观察过程，可执行下面这组推荐命令。

先启动持续业务流：

```bash
h2 iperf -s -p 5001 >/tmp/h2_5001.log &
h4 iperf -s -p 5002 >/tmp/h4_5002.log &
h4 iperf -s -p 5003 >/tmp/h4_5003.log &
h2 iperf -s -p 5004 >/tmp/h2_5004.log &
h1 ping 10.0.0.2 >/tmp/h1_h2_ping.log &
h3 ping 10.0.0.4 >/tmp/h3_h4_ping.log &
h1 iperf -c 10.0.0.2 -p 5001 -t 45 >/tmp/h1_h2_iperf.log &
h3 iperf -c 10.0.0.4 -p 5002 -t 45 >/tmp/h3_h4_iperf.log &
h1 sh -c 'sleep 8; iperf -c 10.0.0.4 -p 5003 -t 16 >/tmp/h1_h4_iperf.log' &
h3 sh -c 'sleep 16; iperf -c 10.0.0.2 -p 5004 -t 16 >/tmp/h3_h2_iperf.log' &
```

上述 3 条命令都放到后台执行后，`Mininet CLI` 仍然可继续输入命令。此时在业务流运行过程中，模拟核心链路 `s4-s6` 故障与恢复：

```bash
link s4 s6 down
link s4 s6 up
```

建议做法：

- 先执行 `link s4 s6 down`。
- 等待 5 到 10 秒，观察控制器日志中的 `Link del` 和重路由输出。
- 再执行 `link s4 s6 up`。
- 再等待 5 到 10 秒，观察控制器日志中的 `Link add` 和路径恢复输出。

实验结束后，可在 `Mininet CLI` 中查看结果：

```bash
h1 tail -n 8 /tmp/h1_h2_iperf.log
h3 tail -n 8 /tmp/h3_h4_iperf.log
h1 tail -n 8 /tmp/h1_h4_iperf.log
h3 tail -n 8 /tmp/h3_h2_iperf.log
h1 tail -n 5 /tmp/h1_h2_ping.log
h3 tail -n 5 /tmp/h3_h4_ping.log
```

说明：

- `multiflow` 场景下同时存在长流和竞争突发流，学习信号会明显强于旧版 2 主机场景。
- `s4-s6` 是拓扑中的优质出口链路，拉断后业务应切换到 `s5` 方向的备用路径。
- 控制器会记录 `Link del` / `Link add` 日志，并立即清理旧路径流表、重算并重装当前主机对的转发路径。
- 监控与 RL 决策循环仍然按 5 秒周期运行，但链路事件本身不再需要等到下一个周期才触发路径重装。
- 动态链路实验产生的控制器日志和指标仍会写入 `/home/ubuntu/毕设/metrics/`。
- 自动演示默认会同时启动 `h1->h2`、`h3->h4` 长流和 `h1->h4`、`h3->h2` 竞争突发流。

### 5.5 持续训练与周期评估

如果希望把 RL 从“单轮在线试跑”升级成“跨多轮持续训练”，推荐使用下面这组参数。

训练控制器：

```bash
cd /home/ubuntu/毕设
source .venv/bin/activate
TOPOLOGY_SCENARIO=multiflow \
RL_AGENT_MODE=train \
RL_BATCH_SIZE=8 \
RL_MIN_REPLAY_SIZE=8 \
RL_CHECKPOINT_SAVE_INTERVAL=5 \
RL_EVAL_INTERVAL_ACTIVE_STEPS=50 \
RL_EVAL_WINDOW_ACTIVE_STEPS=10 \
RL_DECISION_HOLD_STEPS=2 \
RL_REWARD_MAX_UTIL_GAIN_WEIGHT=0.7 \
RL_REWARD_MEAN_UTIL_GAIN_WEIGHT=0.25 \
RL_REWARD_UTIL_DELTA_GAIN_WEIGHT=0.1 \
RL_REWARD_OVERLOAD_THRESHOLD=0.8 \
RL_REWARD_OVERLOAD_PENALTY_WEIGHT=1.2 \
RL_REWARD_HOP_PENALTY_WEIGHT=0.05 \
RL_REWARD_CHURN_PENALTY_WEIGHT=0.05 \
ryu-manager control_plane/ryu_main.py control_plane/network_monitor.py
```

训练流量：

```bash
cd /home/ubuntu/毕设
sudo python3 topology/mininet_topo.py \
  --scenario multiflow \
  --auto-dynamic-demo \
  --demo-repeat 8 \
  --demo-repeat-gap 3 \
  --exit-after-demo
```

说明：

- `--demo-repeat 8` 会把自动动态实验连续跑 8 轮，用同一个控制器进程持续累积 active 样本。
- `RL_BATCH_SIZE=8` 和 `RL_MIN_REPLAY_SIZE=8` 可以让训练在前几轮就开始出现非空 `loss`，不再卡在“样本不够 32 条”。
- `RL_CHECKPOINT_SAVE_INTERVAL=5` 会按训练步数周期性保存 checkpoint，避免控制器退出后重新从头探索。
- `RL_EVAL_INTERVAL_ACTIVE_STEPS=50` 与 `RL_EVAL_WINDOW_ACTIVE_STEPS=10` 会在训练过程中定期插入 10 个 greedy 评估步，用于观察当前策略是否真的变好。
- `RL_DECISION_HOLD_STEPS=2` 会让同一个动作至少维持 2 个控制周期，减小频繁切换路径带来的抖动。
- reward 现在支持通过环境变量调权重。上面的推荐值会更强地惩罚峰值过载、同时放松对 path churn 的抑制，让策略更愿意主动分流，而不是顶着热点长时间维持接近 static 的选路。
- 新日志里会额外记录 `agent_phase`、`agent_mode`、`train_steps`、`replay_size` 和 `checkpoint_saved` 字段，便于区分训练步和评估步。

如果你只想评估已经训练好的模型，而不继续训练，可执行：

```bash
cd /home/ubuntu/毕设
source .venv/bin/activate
TOPOLOGY_SCENARIO=multiflow \
RL_AGENT_MODE=eval \
ryu-manager control_plane/ryu_main.py control_plane/network_monitor.py
```

再用同样的 Mininet 自动实验命令压流即可。`eval` 模式会加载对应场景的 checkpoint，并以 `epsilon=0` 的 greedy 方式选动作，不会继续写入经验回放或更新参数。

## 6. 系统运行过程说明

系统运行时的内部流程如下：

1. `Mininet` 中的交换机通过 OpenFlow 1.3 与 `Ryu` 控制器建立连接。
2. 控制器发现交换机和链路后，维护网络拓扑图。
3. 监控模块定期采集交换机端口统计信息，计算链路带宽利用率。
4. 控制器根据当前链路利用率构造状态向量。
5. 强化学习智能体根据状态选择一组链路权重参数。
6. 最短路径模块基于当前权重重新计算转发路径。
7. 当链路新增、删除或恢复时，控制器会立即清理旧路径流表并按当前拓扑重装路径。
8. 流表安装模块将新路径下发到交换机。
9. 当网络中继续产生流量时，控制器会再次采样状态并重复上述过程。

## 7. 可视化结果生成

如果需要根据真实实验日志生成学习曲线图和性能对比图，可执行：

```bash
cd /home/ubuntu/毕设
source .venv/bin/activate
python visualization/plotter.py
```

执行后会在当前目录生成：

- `learning_curve.png`
- `comparison.png`

说明：

- `learning_curve.png` 默认读取 `metrics/` 下最新的 RL 日志。
- `comparison.png` 默认对比 `metrics/` 下最新的 `rl_metrics_*.jsonl` 和 `static_metrics_*.jsonl`。
- 如果只做了 RL 实验、还没有静态基线日志，脚本会跳过对比图，仅生成学习曲线。
- 若要显式指定日志文件，可使用：

```bash
python visualization/plotter.py \
  --rl-log metrics/rl_metrics_时间戳.jsonl \
  --baseline-log metrics/static_metrics_时间戳.jsonl \
  --metric mean_utilization
```

在 `Mininet CLI` 中退出：


```bash
sudo mn -c
```

## 9. 注意事项

- 必须先启动控制器，再启动拓扑，否则交换机无法正常接入控制器。
- 运行 Mininet 时通常需要 `sudo` 权限。
- `agent/train.py` 当前主要提供智能体实现和示例训练桩，完整实验闭环主要由 `Ryu` 控制器内部调用，不需要单独运行。
- 当前控制器对未知目的地址只向接入口定向泛洪，用于避免环形拓扑里的广播风暴。
- 若上一次实验未正常退出，优先执行 `sudo mn -c` 进行清理。
