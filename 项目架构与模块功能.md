# 项目架构与模块功能

## 1. 项目定位

本项目是一个基于强化学习的 SDN 路由实验系统，核心目标是：

- 使用 `Mininet` 构造可控拓扑与业务流量。
- 使用 `Ryu` 采集链路状态、维护拓扑并下发流表。
- 使用 `PyTorch` 中的 DQN 智能体动态选择链路权重策略。
- 在动态链路故障/恢复和多流竞争场景下，对比 `RL` 与 `static` 的路由表现。

## 3. 运行时主链路

系统运行时的主流程如下：

1. `topology/mininet_topo.py` 启动 Mininet，创建交换机、主机与链路，并可自动发起动态实验。
2. `control_plane/ryu_main.py` 作为 Ryu 主控制器启动，发现交换机、链路与主机。
3. `control_plane/network_monitor.py` 周期采集端口统计信息，换算链路速率和带宽利用率。
4. `control_plane/rl_signal.py` 提供动作空间、reward 计算和路径 churn 计算逻辑。
5. `agent/train.py` 中的智能体根据状态做动作选择，并在训练模式下执行经验回放与 checkpoint 保存。
6. `control_plane/shortest_path.py` 基于当前动作对应的链路权重计算最短路径。
7. `control_plane/flow_installer.py` 将路径下发为交换机流表。
8. `control_plane/ryu_main.py` 把 reward、utilization、action、loss 等指标写入 `metrics/`。
9. `visualization/plotter.py` 读取日志生成学习曲线和 RL/static 对比图。

## 4. 各目录与模块功能

### 4.1 `topology/`

`topology/topology_catalog.py`

- 定义 `HostSpec`、`SwitchLinkSpec`、`FlowSpec` 三类场景数据结构。
- 维护 `legacy`、`multiflow`、`complexflow` 三套实验场景。
- 提供场景查询、链路目录构造、主机映射和主机对数量计算等基础函数。

`topology/mininet_topo.py`

- 创建 Mininet 拓扑并连接远端 Ryu 控制器。
- 自动启动 `iperf` 服务端/客户端与 `ping` 探测流。
- 支持自动动态实验：
  - 指定故障链路
  - 自动执行 `link down / link up`
  - 支持多轮重复实验
- 支持实验后自动退出，便于批量训练与评估。

### 4.2 `control_plane/`

`control_plane/ryu_main.py`

- 项目主控制器，负责串联整个控制平面。
- 主要职责：
  - 维护交换机、链路、主机与邻接关系
  - 构造 RL 状态向量
  - 根据 `RL_ROUTING_MODE` 选择 `rl` 或 `static`
  - 根据 `RL_AGENT_MODE` 选择 `train` 或 `eval`
  - 在链路变化时触发重路由与流表重装
  - 记录 metrics 和保存 checkpoint
- 这是项目中最核心的调度入口。

`control_plane/network_monitor.py`

- 向交换机周期请求端口描述与端口统计信息。
- 维护端口容量、速率和带宽利用率缓存。
- 为 `ryu_main.py` 提供网络状态数据。

`control_plane/rl_signal.py`

- 定义离散动作空间 `ROUTING_PROFILES`。
- 根据链路利用率、容量、时延计算链路权重。
- 计算 reward、overload penalty、path churn 等训练信号。
- 通过环境变量支持 reward 权重调参。

`control_plane/shortest_path.py`

- 封装基于 Dijkstra 的最短路径计算。
- 支持有向链路权重，适配当前控制器中的链路模型。

`control_plane/flow_installer.py`

- 将控制器计算出的路径转成 OpenFlow 流表安装逻辑。
- 负责每跳转发端口解析和最终主机口输出。

### 4.3 `agent/`

`agent/rl_model.py`

- 定义 `RoutingDQN` 神经网络结构。
- 输入为控制器构造的状态向量，输出为离散动作的 Q 值。

`agent/train.py`

- 定义 `AgentConfig` 与 `Agent`。
- 实现：
  - epsilon-greedy 动作选择
  - 经验回放
  - target network 更新
  - checkpoint 保存与恢复
- 当前版本只保留项目真实使用到的 checkpoint 接口，旧的兼容加载/保存桩代码已清理。

### 4.4 `visualization/`

`visualization/plotter.py`

- 从 `metrics/*.jsonl` 中读取实验日志。
- 生成学习曲线图。
- 生成 RL 与 static 的指标对比图。
- 支持切换 `mean_utilization`、`max_utilization`、`path_churn` 等不同指标。

### 4.5 `tests/`

`tests/test_agent.py`

- 验证经验回放触发条件。
- 验证 checkpoint 保存/恢复是否能完整恢复训练状态。

`tests/test_rl_signal.py`

- 验证动作空间数量。
- 验证 reward 方向性、overload 惩罚和 churn 偏好。
- 验证最短路径与场景目录函数行为。

## 5. 关键运行配置

最常用的环境变量如下：

- `TOPOLOGY_SCENARIO`
  - 选择实验场景，如 `multiflow`、`complexflow`。
- `RL_ROUTING_MODE`
  - `rl` 或 `static`。
- `RL_AGENT_MODE`
  - `train` 或 `eval`。
- `RL_CHECKPOINT_PATH`
  - 显式指定模型文件路径；未指定时走默认路径 `checkpoints/<scenario>_rl_agent.pt`。
- `RL_REWARD_*`
  - 控制 reward 中 utilization gain、overload penalty、hop penalty、churn penalty 等权重。

## 6. 运行产物说明

`checkpoints/`

- 保存按场景归档的模型文件。
- 当前目录应理解为“可复用模型入口”，而不是长期堆积历史快照的归档仓库。

`metrics/`

- 保存最新实验日志。
- 若只关心当前实验，保留最近一轮结果即可；旧日志可按需清理。

根目录下的 `*.png`

- 属于日志绘图后的可视化结果。
- 可随实验重跑而重新生成。
